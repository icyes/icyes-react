# PulseOps：React 企业级实战项目需求与学习路线

> 文档版本：v1.0  
> 制定日期：2026-07-03  
> 当前状态：待启动  
> 建议周期：12～16 周，每周 8～12 小时；时间不足时按验收结果推进，不按日历赶进度

## 1. 最终目标

完成一个可演示、可测试、可部署、可讲解的企业级 React 项目，并形成以下能力：

1. 能独立把产品需求拆成页面、组件、状态、接口和测试任务。
2. 理解 React 的渲染模型、状态管理、Effect、Hooks、组件组合与性能边界。
3. 熟练使用 TypeScript、路由、服务端状态、表单、权限、测试和工程工具。
4. 能遵循分支、提交、Code Review、CI、发布和故障复盘等团队流程。
5. 面试时能讲清技术选型、复杂功能设计、性能优化、质量保障和取舍。
6. 进入大型前端项目后，能快速定位入口、数据流、公共层和业务模块。

这不是一个“堆功能”的后台模板。每个阶段都必须通过验收、测试和复盘，才进入下一阶段。

## 2. 实战项目需求

### 2.1 项目名称

**PulseOps — 企业协作与工单管理平台**

### 2.2 业务背景

一家中型互联网公司的产品、研发、测试和客服团队，需要统一管理客户问题、内部任务和线上故障。当前问题散落在聊天记录和表格中，负责人不清晰、处理进度不可追踪、管理者无法判断瓶颈。

PulseOps 需要提供工单全生命周期管理、团队协作、权限控制、数据看板和操作审计。

### 2.3 用户角色

| 角色 | 核心诉求 | 主要权限 |
| --- | --- | --- |
| 普通成员 | 创建、查看和处理自己的工单 | 创建工单、评论、修改本人负责的工单 |
| 项目管理员 | 管理团队工作和流程 | 分配负责人、变更状态、管理成员与标签 |
| 只读访客 | 查看项目进展 | 只读访问被授权的项目 |
| 系统管理员 | 管理平台和审计风险 | 用户、角色、全局配置、审计日志 |

### 2.4 核心业务对象

- 用户 User、团队 Team、项目 Project
- 工单 Ticket、评论 Comment、附件 Attachment
- 标签 Label、通知 Notification、活动记录 Activity
- 角色 Role、权限 Permission、审计日志 AuditLog

### 2.5 核心业务规则

1. 工单编号在项目内唯一，格式为 `项目标识-序号`，例如 `WEB-1024`。
2. 工单状态为：待处理、处理中、待验证、已完成、已关闭。
3. 非法状态流转必须被前端阻止，并由服务端再次校验。
4. 工单必须有标题、项目、优先级和创建人；负责人可以为空。
5. 删除采用软删除，高风险操作必须二次确认并写入审计日志。
6. 用户只能看到被授权项目；按钮权限和路由权限都要校验。
7. 列表筛选条件需要同步到 URL，刷新、前进后退和分享链接后仍可恢复。
8. 评论支持乐观更新；失败时回滚并给出可重试提示。

### 2.6 功能范围

#### P0：必须完成

- 登录、退出、登录态恢复、401 统一处理
- 应用主框架、菜单、面包屑、个人设置、404/403 页面
- 工单列表：分页、排序、组合筛选、搜索、空态、错误态
- 工单详情：基础信息、状态流转、负责人、标签、活动时间线
- 创建/编辑工单：校验、草稿保护、提交防重复、错误定位
- 评论：发布、删除、失败回滚
- RBAC 路由与操作权限
- 仪表盘：关键指标、趋势图、状态分布
- 单元/组件测试、关键 E2E、CI 质量门禁

#### P1：应当完成

- 看板拖拽与乐观更新
- 通知中心与未读数
- 文件上传：类型/体积校验、进度、失败重试
- 审计日志与高级筛选
- 主题切换、国际化、响应式适配
- 错误监控、性能指标与版本信息

#### P2：进阶挑战

- SSE/WebSocket 实时通知
- 大列表虚拟滚动
- 离线草稿与网络恢复
- 微前端方案调研与最小验证（只做验证，不强行改造主项目）
- 前端 BFF 或契约代码生成

### 2.7 非功能要求

- TypeScript 开启严格模式，业务代码禁止随意使用 `any`。
- 常用页面满足键盘操作和基本 WCAG AA 要求。
- 首屏、路由切换和大列表有可测量的性能预算。
- 所有请求具有加载、空数据、失败、重试和权限不足状态。
- 核心逻辑有测试；关键用户路径有独立、稳定的端到端测试。
- 生产构建无 TypeScript 和 ESLint 错误，无敏感配置进入仓库。
- 桌面端优先，同时保证 375px 宽度下核心流程可用。

## 3. 技术栈与选型原则

具体版本在初始化时锁定当日稳定版，不在计划文档中写死版本号。

| 领域 | 计划选型 | 学习目的 |
| --- | --- | --- |
| 基础 | React + TypeScript + Vite | 掌握现代 SPA 的组成和构建过程 |
| 路由 | React Router | 嵌套路由、数据加载、守卫、URL 状态 |
| 服务端状态 | TanStack Query | 缓存、失效、取消、重试、乐观更新 |
| 客户端状态 | `useState` / `useReducer` / Context，后期引入 Zustand | 先理解原理，再判断何时需要状态库 |
| UI | Ant Design + CSS Modules + Design Token | 贴近企业后台，同时练习主题与样式隔离 |
| 表单 | React Hook Form + Zod | 表单性能、类型安全和运行时校验 |
| 请求 | 基于 `fetch` 的类型化请求层 | 理解请求封装、错误模型和取消机制 |
| 接口模拟 | MSW | 前后端并行、异常场景和稳定测试 |
| 单元/组件测试 | Vitest + React Testing Library | 面向用户行为测试，避免实现细节耦合 |
| 端到端测试 | Playwright | 验证跨页面关键链路与浏览器行为 |
| 质量 | ESLint + Prettier + Husky + lint-staged | 把规范变成自动化门禁 |
| 可观测性 | Error Boundary + Web Vitals + 可替换监控适配层 | 错误、性能和版本追踪 |
| CI/CD | GitHub Actions（可按仓库平台替换） | 每次合并前自动检查和构建 |

选型纪律：一个问题只保留一个主方案。学习分支可比较 Redux Toolkit 等方案，但主干不同时堆叠多个同类库。

## 4. 目标目录架构

```text
src/
  app/             # 应用入口、Provider、路由和全局配置
  pages/           # 路由页面，只做页面编排
  features/        # 按业务能力组织：auth、tickets、dashboard...
  entities/        # 稳定领域对象及其通用展示
  shared/
    api/           # 请求客户端、错误模型、接口契约
    components/    # 无业务含义的公共组件
    hooks/         # 通用 Hooks
    lib/           # 纯函数与第三方适配
    styles/        # 全局样式和设计 Token
    types/         # 跨业务公共类型
  mocks/           # MSW handlers 与测试数据工厂
tests/
  e2e/             # Playwright 端到端测试
docs/
  decisions/       # ADR 技术决策记录
  requirements/    # 需求与验收说明
```

依赖方向：`app/pages → features → entities → shared`。下层不能反向依赖上层；业务模块之间优先通过公开入口和清晰的数据契约协作。

## 5. 大型团队开发流程

每个小目标都使用同一套闭环：

1. **需求澄清**：写用户故事、边界、非目标和验收标准。
2. **任务拆分**：每项任务尽量控制在半天到两天，标注风险和依赖。
3. **技术设计**：复杂任务先写简短 ADR，说明选择、备选方案和代价。
4. **开发**：从 `main` 创建 `feat/<ticket-id>-<topic>` 分支，小步提交。
5. **自测**：类型检查、Lint、单元/组件测试、关键手工路径。
6. **Code Review**：提交 PR，描述背景、改动、截图、测试方法和风险。
7. **持续集成**：所有门禁通过后才能合并，禁止“先合再修”。
8. **发布与复盘**：记录结果、缺陷、指标和下一轮改进。

### Definition of Ready：允许开工前

- [ ] 用户价值和范围清晰
- [ ] 验收标准可验证
- [ ] 接口契约或 Mock 已明确
- [ ] 依赖、风险和非目标已记录

### Definition of Done：允许完成前

- [ ] 功能满足验收标准，异常和权限分支已覆盖
- [ ] 类型检查、Lint、测试和生产构建全部通过
- [ ] 新增逻辑有合适层级的测试
- [ ] 无障碍、响应式和错误提示经过检查
- [ ] PR 自检完成，相关文档同步更新
- [ ] 能在 3 分钟内解释实现、权衡和一个待改进点

## 6. 分阶段学习路线

每阶段的“出口验收”必须完成。建议时间只是节奏参考。

### M0：工程启动与基线（2～3 天）

**交付目标**：任何成员克隆仓库后，按 README 可在 10 分钟内启动并完成所有检查。

小目标：

- [ ] 使用 Vite 初始化 React + TypeScript 项目，理解入口、HTML、模块和构建产物
- [ ] 配置严格 TypeScript、路径别名、ESLint、Prettier 和 EditorConfig
- [ ] 建立开发/测试/生产环境变量示例，明确哪些变量会进入浏览器
- [ ] 加入提交前检查、统一脚本和基础 CI
- [ ] 创建 PR 模板、Issue 模板、贡献指南和首个 ADR
- [ ] 完成应用壳：顶栏、侧栏、内容区、错误边界和基础路由

React 重点：JSX、组件、Props、事件、条件渲染、列表与 `key`。  
出口验收：`dev`、`build`、`lint`、`typecheck`、`test` 均可运行；首页和 404 可访问。

### M1：React 基础与可复用 UI（1 周）

**交付目标**：实现仪表盘静态版及一组可复用反馈组件。

小目标：

- [ ] 用组件组合拆分仪表盘，避免巨型组件
- [ ] 实现 Loading、Empty、ErrorState、ConfirmDialog
- [ ] 使用受控组件完成筛选条
- [ ] 用 `useState`、状态提升和派生值管理交互
- [ ] 为纯函数与反馈组件编写首批测试
- [ ] 使用 React DevTools 观察渲染原因

React 重点：状态快照、批处理、不可变更新、组合优于继承。  
出口验收：能画出组件树和状态归属，并解释为什么派生数据通常不应另存为 state。

### M2：路由、布局与 URL 状态（1 周）

**交付目标**：搭好所有 P0 页面骨架，工单筛选可通过链接分享。

小目标：

- [ ] 嵌套路由、动态参数、默认页、404 和懒加载
- [ ] 菜单、标题、面包屑与路由配置保持单一数据源
- [ ] 将分页、搜索、排序和筛选同步到 URL
- [ ] 处理非法参数并提供安全默认值
- [ ] 验证刷新、前进、后退和复制链接

React 重点：组件生命周期、路由边界、Suspense 基础。  
出口验收：复制带筛选条件的 URL 到新窗口，页面状态完全恢复。

### M3：数据请求与服务端状态（1～1.5 周）

**交付目标**：接入 MSW 契约，完成真实交互的工单列表和详情。

小目标：

- [ ] 设计统一响应、分页、业务错误和字段错误模型
- [ ] 封装类型化请求客户端，支持鉴权头、超时、取消和错误转换
- [ ] 使用 Query Key Factory 管理缓存键
- [ ] 完成加载、空态、错误、重试、预取和分页占位体验
- [ ] 处理竞态、重复请求、缓存新鲜度和请求取消
- [ ] 编写成功、空数据、500、慢请求的 Mock 场景

React 重点：Effect 只负责与外部系统同步；区分客户端状态与服务端状态。  
出口验收：能解释为什么不把接口数据默认塞进全局 Store，以及缓存何时失效。

### M4：复杂表单与数据校验（1 周）

**交付目标**：完成创建/编辑工单流程。

小目标：

- [ ] 基础字段、动态字段、联动字段和异步选项
- [ ] 前端 Schema 校验与 TypeScript 类型推导
- [ ] 映射服务端字段错误到具体控件
- [ ] 离开未保存页面时提示，提交期间防重复
- [ ] 编辑态回填、脏状态、重置和提交成功跳转
- [ ] 覆盖正常提交、校验失败和服务端失败测试

React 重点：受控/非受控组件、Ref、闭包、事件边界。  
出口验收：弱网和重复点击下不会创建重复数据，错误能准确落到字段。

### M5：身份认证与 RBAC 权限（1 周）

**交付目标**：不同角色看到并能执行的操作符合权限矩阵。

小目标：

- [ ] 登录、退出、初始化恢复和 Token 失效处理
- [ ] 路由级权限、组件级权限和数据级权限
- [ ] 实现 `Can` 权限组件，但核心判断保持为可测试纯函数
- [ ] 处理 401、403、会话过期和登录后返回原页面
- [ ] 避免把前端隐藏按钮误当成安全边界

React 重点：Context 使用边界、自定义 Hook、Provider 组合。  
出口验收：权限矩阵测试通过；直接输入无权 URL 时得到正确反馈。

### M6：状态设计与跨模块协作（1 周）

**交付目标**：完成通知中心、个人偏好和跨页面草稿。

小目标：

- [ ] 制作“状态分类表”：URL、服务端、表单、本地 UI、跨页面客户端状态
- [ ] 先用 Context + Reducer 实现，再识别性能和维护问题
- [ ] 只在确有收益时引入 Zustand，并记录 ADR
- [ ] 使用选择器减少无关订阅更新
- [ ] 完成 Store 单元测试与刷新后的持久化策略

React 重点：Reducer、Context、外部 Store、状态归一化。  
出口验收：能为任意新状态判断它该放在哪里，并说出依据。

### M7：复杂交互与并发场景（1～1.5 周）

**交付目标**：完成看板、评论和文件上传。

小目标：

- [ ] 看板拖拽、状态规则校验和键盘替代操作
- [ ] 乐观更新、失败回滚和并发冲突提示
- [ ] 评论临时 ID、重复提交和删除确认
- [ ] 上传类型/体积校验、进度、取消与重试
- [ ] 对高频搜索使用防抖或 React 并发特性，并测量收益

React 重点：Transition、乐观 UI、稳定标识、竞态处理。  
出口验收：模拟 30% 请求失败时，UI 不丢数据、不假成功、可恢复。

### M8：测试体系（1～1.5 周）

**交付目标**：建立可持续的测试金字塔，而不是追求虚高覆盖率。

小目标：

- [ ] 纯业务规则使用单元测试
- [ ] 组件使用接近用户操作的查询与断言
- [ ] MSW 覆盖网络成功和失败，不 Mock 组件内部实现
- [ ] Playwright 覆盖登录、建单、筛选、状态流转和权限五条主链路
- [ ] 测试数据相互隔离，禁止依赖执行顺序
- [ ] CI 上传失败截图、报告和 Trace

质量基线：核心领域纯函数分支覆盖率 ≥ 90%；全项目覆盖率作为趋势指标，初始门槛 70%，不以无意义测试凑数。  
出口验收：任意测试可单独、重复运行；重构组件结构但行为不变时测试基本不受影响。

### M9：性能、无障碍与体验（1 周）

**交付目标**：用数据发现和修复瓶颈，完成基础无障碍审计。

小目标：

- [ ] React Profiler 定位一次真实重复渲染并记录前后数据
- [ ] 路由级拆包、图表懒加载和资源预取
- [ ] 大列表达到阈值后引入虚拟滚动
- [ ] 谨慎使用 `memo`、`useMemo`、`useCallback`，每次优化有证据
- [ ] 处理焦点、标签、语义结构、对比度和键盘导航
- [ ] 建立 Bundle 与 Web Vitals 预算

建议预算：关键页面桌面端生产包首屏 JS gzip 目标 ≤ 250 KB（不含按需异步块）；测试环境 LCP ≤ 2.5s、CLS ≤ 0.1、INP ≤ 200ms。若环境不同，在 ADR 中调整并解释。  
出口验收：提交一份性能前后对比和无障碍检查记录。

### M10：可靠性、安全与可观测性（1 周）

**交付目标**：应用出错时可发现、可定位、可降级。

小目标：

- [ ] 页面级 Error Boundary、请求错误分类和用户可恢复操作
- [ ] 错误监控适配层记录版本、路由、用户匿名标识和关联 ID
- [ ] Web Vitals 上报与关键操作埋点
- [ ] 防范 XSS、开放重定向、敏感日志和依赖风险
- [ ] Feature Flag 控制高风险功能
- [ ] 编写一个故障演练和复盘文档

出口验收：注入页面异常与接口异常后，用户有合理反馈，开发者能定位版本和链路。

### M11：交付、文档与面试演练（1 周）

**交付目标**：完成线上演示、项目文档和可复述的项目故事。

小目标：

- [ ] CI 执行安装、Lint、类型、测试、构建和 E2E
- [ ] 建立预览环境与生产发布流程，明确回滚方式
- [ ] README 包含架构、启动、脚本、测试和部署说明
- [ ] 输出架构图、权限矩阵、接口约定和 3～5 个 ADR
- [ ] 整理性能、测试、复杂交互和故障处理四类项目案例
- [ ] 完成两轮模拟 Code Review 和两轮模拟面试

出口验收：陌生开发者只看文档可启动项目；能在 5 分钟内完成项目介绍并接受追问。

## 7. 每周固定节奏

| 环节 | 产出 | 建议占比 |
| --- | --- | --- |
| 需求与设计 | 用户故事、验收标准、草图或 ADR | 15% |
| 编码实现 | 小步提交的功能代码 | 45% |
| 测试与调试 | 自动测试、边界检查、性能观察 | 20% |
| Review 与重构 | 自检、评审意见、代码改进 | 15% |
| 复盘 | 本周掌握、踩坑、下周行动 | 5% |

遇到问题时遵循：先描述现象和预期 → 缩小复现范围 → 查看错误/网络/状态 → 提出两个假设 → 用实验验证。不要直接复制答案。

## 8. 第一个可执行迭代：Sprint 0

Sprint 目标：建立工程质量基线，并交付可导航的应用壳。完成后再开始业务功能。

### S0-01 环境与基础能力盘点（30～60 分钟）

- [ ] 确认 Node、包管理器、Git 和编辑器环境
- [ ] 用少量题目自测 JavaScript：闭包、事件循环、数组不可变操作、模块、Promise
- [ ] 记录每天/每周可投入时间

验收：在 `docs/learning-notes/00-baseline.md` 写出当前水平、薄弱点和时间预算。

### S0-02 初始化项目（1～2 小时）

- [ ] 创建 Vite React TypeScript 项目
- [ ] 选择并锁定一种包管理器
- [ ] 整理 `dev/build/preview/typecheck` 脚本
- [ ] 提交第一个可运行版本

验收：新环境按 README 操作可打开首页并完成生产构建。

### S0-03 质量工具（2～3 小时）

- [ ] 开启 TypeScript 严格模式
- [ ] 配置 ESLint、Prettier、EditorConfig
- [ ] 配置提交前只检查暂存文件
- [ ] 禁止未使用变量、危险断言和随意 `any`

验收：故意制造格式、类型和 Lint 错误时，工具能在正确阶段拦截。

### S0-04 测试基线（2～3 小时）

- [ ] 配置 Vitest、React Testing Library 和浏览器 DOM 环境
- [ ] 为首页写一个面向用户行为的测试
- [ ] 配置 Playwright 并完成一个 smoke test

验收：单元/组件测试与 E2E 可分别运行，报告可读。

### S0-05 应用壳与路由（4～6 小时）

- [ ] 实现顶栏、侧栏、内容区和移动端菜单
- [ ] 建立仪表盘、工单列表占位页和 404
- [ ] 实现当前菜单态、面包屑和路由懒加载
- [ ] 加入页面级 Error Boundary

验收：桌面与 375px 宽度均可导航；刷新子路由可正常显示。

### S0-06 团队协作材料（1～2 小时）

- [ ] 新增 PR 模板和贡献指南
- [ ] 写 `0001-frontend-stack.md` ADR
- [ ] 创建 CI，执行类型、Lint、测试与构建
- [ ] 按 PR 模板完成 Sprint 0 自我评审

验收：CI 全绿，PR 描述含截图、测试证据、风险和后续事项。

## 9. 面试能力映射

| 项目经历 | 需要能回答的追问 |
| --- | --- |
| 状态分类 | 服务端状态和客户端状态有何不同？Context 为什么不适合所有全局状态？ |
| 请求缓存 | Query Key 如何设计？失效、预取、取消、重试分别何时使用？ |
| Effect | 哪些逻辑不该写在 Effect？如何处理闭包、竞态与清理？ |
| 权限系统 | 路由、按钮、数据权限如何分层？为什么前端权限不是安全边界？ |
| 复杂表单 | 如何做类型安全、联动、服务端错误映射、草稿和性能优化？ |
| 乐观更新 | 如何生成临时状态、回滚、处理重复提交和并发冲突？ |
| 性能 | 如何证明优化有效？React 重渲染与 DOM 更新有什么区别？ |
| 测试 | 测什么、不测什么？如何降低脆弱测试与 E2E 偶发失败？ |
| 工程化 | CI 为什么这样排序？如何做依赖治理、发布与回滚？ |
| 架构 | 为什么按业务能力组织？如何控制依赖方向和公共层膨胀？ |

每完成一个阶段，至少形成一个 STAR 故事：背景、任务、行动、结果；其中“结果”优先使用缺陷数、耗时、包体积、渲染次数、覆盖率等可验证数据。

## 10. 学习记录与评审机制

建议每个阶段保留以下材料：

```text
docs/learning-notes/
  00-baseline.md
  M1-retrospective.md
  M2-retrospective.md
  ...
```

每次复盘回答五个问题：

1. 这次交付给用户带来了什么？
2. 我能不用资料解释哪三个概念？
3. 哪个 Bug 最有价值，根因是什么？
4. 哪项设计是权衡，而不是唯一正确答案？
5. 下一阶段只改进一件事，会是什么？

指导方式采用逐级提示：先让学习者给出方案 → 用问题暴露盲区 → 提供最小示例 → 一起 Review 完整实现。关键代码由学习者实际输入和解释，避免“看懂了等于会了”。

## 11. 官方依据与延伸阅读

- React 官方已弃用 Create React App；学习型自建应用可从构建工具开始：[React Installation](https://react.dev/learn/installation)
- Effect 是连接外部系统的逃生舱，不应承担可在渲染或事件中完成的逻辑：[You Might Not Need an Effect](https://react.dev/learn/you-might-not-need-an-effect)
- TypeScript 的目标是在运行前发现类型错误：[TypeScript Handbook](https://www.typescriptlang.org/docs/handbook/intro.html)
- TanStack Query 面向异步服务端状态、缓存和同步：[TanStack Query React Docs](https://tanstack.com/query/latest/docs/framework/react/overview)
- Testing Library 推荐从用户可观察行为出发，避免依赖实现细节：[Testing Library Introduction](https://testing-library.com/docs/)
- Playwright 推荐测试用户可见行为、保持测试隔离并使用面向用户的定位方式：[Playwright Best Practices](https://playwright.dev/docs/best-practices)
